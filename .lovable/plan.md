# План: сделать CRM рабочей на Beget без Supabase и Lovable-хостинга

## Принято как факт

- Lovable-домен в России не подходит.
- Supabase в России без VPN не работает.
- Российский домен и хостинг должны быть на Beget.
- AI-провайдера пока не меняем — оставляем текущую AI-логику и текущие модели/ключи.

## Что показал анализ проекта

Сейчас CRM сильно завязана на Supabase:

- 53 таблицы в базе данных.
- 392 обращения к Supabase в коде.
- 29 серверных функций:
  - AI-технолог
  - AI-сметчик
  - AI-аналитик
  - AI-консультант
  - голосовой помощник
  - Telegram/WhatsApp
  - генерация документов
  - отправка PDF/email
  - webhooks
- Авторизация, роли, права пользователей, файлы и документы тоже завязаны на Supabase.

Поэтому правильное решение с минимальными затратами — **не переписывать весь фронтенд вручную**, а сделать свой backend-слой на PHP/MySQL, который будет заменять Supabase для CRM.

# Целевая схема

```text
Пользователь в России
        ↓
Домен на Beget
        ↓
React CRM из папки dist/
        ↓
/api.php на Beget
        ↓
MySQL на Beget

AI-функции:
React CRM → PHP endpoint → OpenAI / текущий AI-провайдер
```

Фронтенд остаётся React + TypeScript + Vite.

На Beget будет лежать:

```text
/public_html/
  index.html
  assets/
  .htaccess
  api.php
  config.php
  db.php
  auth.php
  functions/
    ai-technical-specialist.php
    ai-estimator.php
    ...
```

# Главная идея миграции

Вместо того чтобы переписывать все страницы CRM, мы делаем **адаптер вместо Supabase**.

Сейчас код вызывает примерно так:

```ts
supabase.from('clients_base').select('*')
supabase.from('tasks').insert(...)
supabase.auth.signInWithPassword(...)
supabase.functions.invoke('ai-technical-specialist', ...)
```

После переноса эти же места должны работать через наш backend:

```text
supabase.from(...)          → POST /api.php?action=query
supabase.auth.login(...)    → POST /api.php?action=login
supabase.functions.invoke() → POST /api.php?action=function
```

То есть мы не переделываем 392 места вручную, а заменяем внутренний клиент Supabase на свой совместимый клиент.

# Этап 3. Сделать авторизацию на PHP

Supabase Auth больше использовать нельзя.

Нужно сделать свою авторизацию:

- регистрация
- вход по email/паролю
- выход
- восстановление пароля можно добавить позже
- хранение пароля через `password_hash()`
- проверка пароля через `password_verify()`
- сессия через secure cookie или JWT
- роли пользователей через отдельную таблицу `user_roles`

Важно: роли нельзя хранить просто в профиле пользователя. Нужна отдельная таблица ролей, как и сейчас.

# Этап 4. Сделать универсальный PHP API для таблиц

Создать `/api.php`, который будет принимать запросы от React.

Минимальные действия:

```text
select
insert
update
delete
single
order
limit
filter by eq
```

Пример запроса от фронтенда:

```json
{
  "table": "tasks",
  "action": "select",
  "filters": [
    { "column": "user_id", "operator": "eq", "value": "..." }
  ]
}
```

PHP должен:

1. Проверить пользователя.
2. Проверить доступ к таблице.
3. Автоматически ограничить данные по `user_id`.
4. Выполнить SQL через prepared statements.
5. Вернуть JSON в формате, удобном для текущего React-кода.

# Этап 5. Заменить Supabase-клиент во фронтенде

Сейчас файл:

```text
src/integrations/supabase/client.ts
```

создаёт реальный Supabase client.

Его нужно заменить на свой клиент-адаптер, который будет поддерживать основные методы:

```ts
supabase.from(table).select()
supabase.from(table).insert()
supabase.from(table).update()
supabase.from(table).delete()
supabase.auth.signInWithPassword()
supabase.auth.signOut()
supabase.auth.getSession()
supabase.auth.onAuthStateChange()
supabase.functions.invoke()
```

Это самый экономный путь, потому что большинство страниц CRM смогут продолжить работать без полного переписывания.

# Этап 7. Перенести AI-технолога и AI-сметчика

Текущие Supabase Edge Functions нужно заменить PHP endpoint-ами.

Например:

```text
supabase/functions/ai-technical-specialist/index.ts
```

станет:

```text
/functions/ai-technical-specialist.php
```

Логика сохранится:

1. Получить пользователя.
2. Получить его материалы из MySQL.
3. Получить его услуги из MySQL.
4. Собрать системный промпт.
5. Отправить запрос в текущий AI-провайдер.
6. Получить JSON.
7. Сохранить ТЗ в `technical_specifications`.
8. Вернуть результат во фронтенд.

То же самое отдельно для AI-сметчика.

AI-провайдера пока не меняем.

# Что будет потеряно или изменится

Чтобы не было иллюзий, важно понимать:

## Realtime

Мгновенные обновления Supabase Realtime исчезнут. Их можно заменить позже обычным обновлением раз в несколько секунд.

## Storage

Файлы, аватары и документы нужно будет хранить на Beget в папке uploads.

## RLS

Supabase RLS исчезнет. Безопасность нужно будет реализовать в PHP:

- нельзя доверять `user_id` из браузера
- все запросы должны проверять текущую сессию
- PHP сам должен добавлять ограничения по пользователю

## Edge Functions

Все функции Supabase нужно будет постепенно переписать на PHP.
